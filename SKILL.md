# 论坛图文帖生成器

将一批照片转化为结构清晰、图文并茂的论坛帖子。适用场景包括但不限于：数码产品开箱、美食探店、旅行游记、车展/活动报道、摄影作品分享。

## 工作流程总览

```
阶段1：分析 & 大纲
  逐张识别照片 → 自动归类 → 联网检索背景 → 生成大纲 → 用户确认

阶段2：撰写 HTML
  按确认大纲撰写 → 应用固化风格 → 输出预览

阶段3：审查打磨
  附审查清单 → 用户在对话中反馈 → 逐条修改 → 刷新预览（循环至定稿）

阶段4：交付输出
  ① 定稿 HTML + 重命名图片（按文章顺序）
  ② 用户说「生成BBCode」→ 输出 BBCode 版本
```

---

## 阶段1：分析照片 & 生成大纲

### 1.1 照片识别

逐张读取用户指定的照片目录中的所有图片，识别每张照片的内容。记录：

- 主体（产品/菜品/车辆/建筑/人物/场景等）
- 拍摄角度（特写/全景/侧面/正面/俯拍等）
- 照片中出现的文字/标签/品牌信息

### 1.2 自动归类

将照片按主题自动分组，常见归类维度：

- **数码开箱**：外包装 → 开箱过程 → 产品全貌 → 细节特写 → 配件 → 使用场景
- **美食探店**：门面环境 → 菜单 → 菜品特写 → 饮品 → 环境细节 → 账单/彩蛋
- **旅行游记**：交通 → 地标全景 → 建筑细节 → 美食 → 人文 → 趣事彩蛋
- **车展报道**：展馆入口 → 重点车辆 → 细节 → 其他车型 → 花絮 → 彩蛋
- **摄影分享**：按拍摄主题/地点/时间分组

### 1.3 联网检索背景

根据照片内容和用户提供的话题背景，联网搜索补充信息：

- 产品型号/规格参数/售价/发布信息
- 餐厅评分/特色菜品/人均消费
- 景点历史/开放时间/门票信息
- 车辆型号/改装背景/相关人物
- 任何需要核实或补充的细节

所有外部信息必须在文中标注"资料来自网络"的免责说明。

### 1.4 生成大纲

根据照片归类结果 + 用户需求 + 搜索到的背景信息，生成结构化大纲：

```
📋 文章大纲提案

标题：[拟定标题]
副标题：[拟定副标题/导语]

章节规划：
1. 引子（图1-X）— 怎么发现/为什么去/第一印象
2. 重点章节（图X-X）— 最核心的内容，照片最多
3. 次要章节（图X-X）— 按逻辑排列
...
N. 彩蛋/结尾（图X-X）— 有趣但非主线的内容

图片分配：共XX张，全部纳入大纲
```

向用户展示大纲，等待确认或调整后再进入阶段2。

**禁止**在大纲未确认的情况下开始撰写。

## 阶段2：撰写 HTML 文章

### 2.1 使用 HTML 模板

读取 `assets/article-template.html` 作为骨架，填充内容。模板包含：

- 顶部导航栏 + 文章标题区（仅用于预览，不输出到 BBCode）
- 响应式正文区：最大宽度 800px，居中，适合桌面和移动端
- 目录区块（TOC）：每章标题 + 图片编号范围
- 图片展示区：每张图片 + 图注
- 页脚

### 2.2 固化风格规则

撰写正文时，严格遵守以下规则：

**叙事口吻**
- 第一人称，普通网友分享视角
- 口语化但不随意，有信息量但不官方
- 避免媒体腔、公关腔、评测机构腔

**标题体系**
- 全文只有一个主标题（H1）
- 章节标题使用 H2，层级不跳跃，不滥用标题样式仅为放大字体
- 相邻标题层级连续

**段落**
- 长短适中、逻辑自成一体
- 不出现只有一句话的段落（除非是过渡/强调且有必要）
- 段落间仅一个换行

**文字强调**
- 用粗体（`<strong>` 或 `**`）做关键词强调，不用双引号做强调或装饰
- 不频繁加粗、不连续加粗、不大面积加粗
- 整句加粗仅用于极少数需要高度突出的总结性句子

**标点与排版**
- 中文与英文/数字之间加一个半角空格（如「iPhone 15 Pro」「2026 年 7 月」）
- 中文引号使用直角引号「」，不用直引号 "；嵌套时用『』
- 省略号用两个半角省略号「……」，不用句号代替
- 日期用汉字格式：`2026年7月17日`，不用横线或斜线
- 不将引号用于强调或装饰（全文尽量少用引号）
- 不用斜线表示并列选项

**专有名词**
- 遵守官方大小写：GitHub、NVIDIA、Sony、macOS
- App / app 不用全大写「APP」
- 首次出现的缩略语/专业术语，括号加注全称

**图片版权**
- 所有照片标注「摄影：作者名」
- 引用外部图片须标注来源

**免责声明**
- 涉及外部资料时，在文中适当位置加入「部分资料来自网络，如有出入欢迎指正」

完整风格参考见 `references/style-rules.md`。

### 2.3 图注格式

每张图片下方紧跟图注，格式：

```
▲ 图X 描述文字
```

- `▲` + 一个空格 + `图X` + 一个空格 + 描述
- 不使用 `｜` 分隔符
- 图注居中、灰色小字、不加粗（图号加粗）
- 描述简洁准确，说明照片内容即可

### 2.4 目录规范

文章正文前必须有目录区块，格式：

```
📋 文章目录
1. 章节名称 — 图1~4
2. 章节名称 — 图5~15
3. 章节名称 — 图16~28
...
N. 章节名称 — 图84~87
```

- 目录锚点链接到对应章节
- 图片编号范围让读者/排版时快速定位
- 目录本身简洁，不展开详细信息

### 2.5 输出预览

撰写完成后，用 `present_files` 展示 HTML 文件，用户可在浏览器中预览。

## 阶段3：审查打磨

### 3.1 审查清单

每次输出 HTML 后，附上审查清单：

```
📋 审查清单

【结构】
☐ 开头叙事风格是否符合要求的视角？
☐ 章节顺序是否合理？
☐ 目录图片编号范围是否正确？

【重点内容】
☐ 用户指定的重点是否在文章核心位置？
☐ 重点内容照片数量/图注是否充分？
☐ 彩蛋/次要内容是否放在正确位置？

【图片】
☐ 全部图片是否都已使用？（共X张）
☐ 每张图片的图注是否准确？
☐ 图片章节分配是否合理？

【文案】
☐ 全文是否顺畅、口语化？
☐ 是否有官方腔/媒体腔的句子？
☐ 引号是否已最小化？
☐ 中英文间是否加了空格？

【技术项】
☐ 视频链接是否已嵌入（如有）？
☐ 外部资料是否已标注来源？
☐ 专有名词大小写是否正确？
```

### 3.2 修改流程

1. 用户直接在对话中逐条反馈（如「图3描述改成XX」「开头精简一下」「第5章移到第2章后面」）
2. 逐条读入意见，直接修改 HTML
3. 改完用 `present_files` 刷新预览
4. 重复直到用户满意

**不需要用户打开 HTML 文件手动编辑**，所有修改在对话中完成。

### 3.3 定稿确认

用户明确表示「可以了」「定稿了」或类似信号后，进入阶段4。

---

## 阶段4：输出交付

### 4.1 重命名图片

将图片按文章中的出现顺序重命名，使其编号与「图X」一一对应：

```
图1 → CMST_01.jpg（原 CMST_09.jpg）
图2 → CMST_02.jpg（原 CMST_06.jpg）
...
图87 → CMST_87.jpg（原 CMST_08.jpg）
```

**重命名策略**：两步法防冲突
1. 先给所有文件加临时前缀（如 `_tmp_`）
2. 再按目标编号逐一重命名

### 4.2 交付清单

| 文件 | 说明 |
|------|------|
| `文章名.html` | 最终定稿 HTML，可浏览器预览 |
| `CMST_01.jpg` ~ `CMST_NN.jpg` | 按文章顺序重命名后的图片 |
| `文章名.bbcode.txt` | BBCode 版本（待用户指令生成） |

### 4.3 BBCode 生成

**仅在用户明确说「生成BBCode」时执行。**

BBCode 格式映射规则见 `references/bbcode-rules.md`。转换要点：

- HTML 导航栏、header 不输出到 BBCode
- `[img]文件名[/img]` 作为图片占位符（待上传后替换为 `[attachimg]ID`）
- 主标题：`[align=center][size=6][b]标题[/b][/size][/align]`
- 章节标题：`[size=6][b]标题[/b][/size]`
- 图注：`[align=center][size=4][color=gray]▲ [b]图X[/b] 描述[/color][/size][/align]`
- 引用/免责：`[quote]内容[/quote]`
- **不使用分隔线**
